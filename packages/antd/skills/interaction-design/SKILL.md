# 交互设计方法论

## 核心原则

> **先设计，后编码。** 在写第一行代码之前，必须对整个功能有完整的设计方案。

AI 的默认行为是"收到需求 → 逐个组件拼合 → 输出代码"，这导致：
- 不同视图同类操作用不同组件（不一致）
- 边缘状态（空态/错误/权限不足）被遗忘
- 交互路径割裂，用户体验不流畅

本 Skill 强制 AI 先走完 5 个设计阶段，输出**设计简报**，用户确认后再开始编码。

---

## 适用范围

**必须使用：**
- 涉及 2 个以上视图/页面的功能
- 包含 CRUD 操作的模块
- 有复杂状态流转的交互（审批、多步骤、权限控制）

**不需要使用：**
- 修改单个组件样式
- 修复 bug
- 给已有页面添加单个字段

---

## 执行流程

收到功能需求后，**不要立即写代码**，按以下 5 个阶段依次思考：

---

### Phase 1：任务分析

理解用户要完成的完整任务。

**必须明确：**

```
1. 核心目标：用户在这个功能里要完成什么？（一句话）
2. 数据操作：读取 / 创建 / 编辑 / 删除 / 审批（哪些组合）
3. 用户角色：是否有多种角色？不同角色的权限差异是什么？
4. 功能边界：这个功能从哪里进入？完成后去哪里？
5. 数据规模：单条记录 / 列表（预计多少条）/ 树形结构
```

**判断功能复杂度：**
```
简单（1个视图）    → 单页表单、单个详情卡片
中等（2-3个视图）  → 列表 + 详情/编辑、分步骤表单
复杂（4+个视图）   → 完整 CRUD 模块、工作台、审批流
```

---

### Phase 2：交互流程设计

用流程图描述用户的完整操作路径。**必须覆盖主路径和所有异常路径。**

**流程图格式：**
```
[入口] → [视图A] → [操作] → [视图B] → [完成]
                       ↓
                  [失败/错误] → [错误处理] → [重试/返回]
```

**必须覆盖的路径：**
```
主路径（Happy Path）：
  用户顺利完成所有步骤的路径

取消路径：
  用户在每一步可以取消/返回，数据如何处理？

错误路径：
  表单校验失败 → 如何提示？
  网络请求失败 → 如何提示？是否允许重试？
  
权限路径：
  无权限时 → 隐藏操作 or 403 页面 or disabled？
```

---

### Phase 3：视图架构

确定需要几个独立视图，以及视图间的关系。

**视图类型选择：**

```
用户需要做什么？
│
├── 浏览/筛选数据列表
│   → 列表页（List Pattern）
│
├── 创建/编辑记录
│   ├── 字段 ≤ 8 个，不离开列表   → Drawer（抽屉）
│   ├── 字段 > 8 个，需要专注填写  → 独立表单页
│   └── 有明确步骤关系            → Steps 分步骤表单页
│
├── 查看记录详情
│   ├── 内容简单（≤ 10 个字段）   → Drawer 或 Modal
│   └── 内容复杂、有子数据         → 独立详情页
│
├── 快速确认操作
│   ├── 一句话能说清楚             → Popconfirm（就地确认）
│   └── 需要说明后果/要求输入      → Modal.confirm
│
└── 数据概览 + 多入口
    → 工作台（Dashboard Pattern）
```

**视图关系表（每个功能必须填写）：**

| 视图名称 | 模式 | 触发方式 | 离开方式 | 对应 Skill |
|---------|------|---------|---------|-----------|
| （示例）用户列表 | List | 直接访问 | 点击行/新建 | page-patterns |
| （示例）新建抽屉 | Form | 点击"新建" | 提交成功/取消 | page-patterns |
| （示例）详情页 | Detail | 点击列表行 | 浏览器返回 | page-patterns |

---

### Phase 4：状态清单

**每个视图必须预先设计所有状态，不允许遗漏。**

对每个视图，逐项确认以下状态的 UI 表现：

```
□ 加载中    → 用什么组件？（Skeleton / Spin / Table loading）
□ 空态-首次 → 没有任何数据时，如何引导用户创建？
□ 空态-筛选 → 搜索无结果时，是否提示清除筛选条件？
□ 正常      → 有数据时的主要界面
□ 错误      → 请求失败时，显示什么？是否有重试按钮？
□ 权限不足  → 无权访问时，隐藏 or 禁用 or 403 页面？
□ 部分权限  → 有些操作无权限时，按钮是隐藏还是 disabled？
```

**状态对应的 antd 组件：**

| 状态 | antd 组件 |
|------|----------|
| 列表加载中 | `<Table loading>` 或 `<Skeleton>` |
| 表单提交中 | `<Button loading>` + 禁用所有字段 |
| 空态（首次使用）| `<Empty image={PRESENTED_IMAGE_SIMPLE}>` + 创建按钮 |
| 空态（搜索无结果）| `<Empty description="没有找到匹配结果">` + 清除筛选按钮 |
| 请求失败 | `message.error` (轻量) / `<Result status="error">` (页面级) |
| 权限不足 | `<Result status="403">` |
| 服务器错误 | `<Result status="500">` |

---

### Phase 5：组件角色分配

对每个视图，指定核心 antd 组件及职责，并做一致性检查。

**格式：**

```
视图名：[视图名称]
  主体组件：<Table> / <Form> / <Descriptions> / ...
  操作入口：<Button type="primary"> 位置：[右上角/footer右侧/...]
  确认操作：<Popconfirm> / <Modal.confirm> / <Modal>
  反馈组件：message.success/error / <Alert> / <Result>
  空态组件：<Empty> / <Skeleton>
```

**一致性规则（必须检查）：**
```
□ 所有视图中，"主要操作"按钮类型一致（type="primary"）
□ 所有 Modal/Drawer 中，"确认"在右，"取消"在左
□ 所有危险操作（删除/禁用）使用 danger 类型
□ 所有成功反馈使用 message.success（不用 notification）
□ 所有加载状态使用相同的 loading 组件类型
```

## 常见交互决策参考

### 编辑操作：Drawer vs 独立页面？

```
选 Drawer：
✅ 字段 ≤ 10 个
✅ 需要对照列表内容编辑
✅ 编辑完成后留在列表

选独立页面：
✅ 字段 > 10 个
✅ 有复杂嵌套结构（子表格、多 Tab）
✅ 编辑需要全屏专注
✅ 有明确的"返回"语义
```

### 确认操作：Popconfirm vs Modal.confirm？

```
选 Popconfirm：
✅ 操作语义简单（删除、禁用）
✅ 一句话能描述后果
✅ 不需要用户输入任何内容

选 Modal.confirm：
✅ 需要说明复杂后果
✅ 需要用户输入确认文字（如输入 "DELETE"）
✅ 需要列出会被影响的关联数据
```

### 反馈方式：message vs notification vs Alert？

```
message.success/error：
✅ 轻量操作（保存、更新、切换状态）
✅ 信息 ≤ 30 字
✅ 2-3 秒后自动消失

notification：
✅ 系统级事件（后台任务完成、有新消息）
✅ 需要用户主动关闭
✅ 有标题 + 正文结构

Alert（内嵌）：
✅ 页面级持久提示（系统维护、权限说明）
✅ 需要一直可见，不自动消失
✅ 有操作按钮（如"去设置"）
```

---

## 反模式：不要这样做

```
❌ 收到需求直接开始写第一个组件
❌ 先写完列表页，再决定编辑用 Drawer 还是 Modal
❌ 所有状态都写完才发现没有空态
❌ 不同页面同类操作用不同组件（A 页面删除用 Popconfirm，B 页面用 Modal）
❌ 弹窗里再套弹窗（Modal 里打开新 Modal）
❌ 在没有确认用户设计意图的情况下生成超过 100 行的 UI 代码
```
